**在 PostgreSQL 中，一旦数据量达到 1,000 行以上，`COPY` 几乎在所有场景下都比 `INSERT`（即使是批量插入）高效得多。**

如果数据量达到 **1万行、10万行或百万行**，`COPY` 的优势将是压倒性的，速度可能快 **5 到 20 倍**。

以下是深层机制分析，帮你理解为什么 `COPY` 更快，以及在什么量级下切换。

---

### 1. 为什么 `COPY` 显著快于 `INSERT`？

#### 批量插入 (`batch insert`) 的本质：

即使你使用了 `addBatch()`，在底层，数据库依然需要处理很多重复工作：

* **SQL 解析**：数据库需要解析 SQL 语句（尽管批量插入减少了次数，但依然有开销）。
* **约束检查**：每一行数据提交时，数据库都要频繁检查约束。
* **事务管理**：每一批次提交都会产生大量的日志记录和事务开销。
* **协议往返**：虽然是批处理，但依然遵循 SQL 协议的请求-响应模型。

#### `COPY` 的本质：

`COPY` 是 PostgreSQL 为了大数据量导入而设计的**专用通道**（Fast Path）：

* **绕过 SQL 解析**：`COPY` 不需要解析复杂的 SQL 语法，它直接接收一个预定义的格式化数据流（CSV 或二进制）。
* **减少日志量**：`COPY` 在写入时能显著减少 WAL（预写日志）的生成量。
* **单向数据流**：它建立了一个连续的数据流，而非一个一个的“请求 $\rightarrow$ 响应”循环，极大地降低了网络 I/O 开销。
* **直接写入**：数据直接进入存储引擎，跳过了许多 SQL 层面的中间处理步骤。

---

### 2. 不同数据量级的性能对比

| 数据量                | 插入方式      | 性能表现      | 建议                                              |
| :-------------------- | :------------ | :------------ | :------------------------------------------------ |
| **1 ~ 100 行**        | 单条 `INSERT` | 极快 (毫秒级) | 直接 `INSERT` 即可，开发最简单。                  |
| **100 ~ 1,000 行**    | `addBatch()`  | 很快          | 批量插入即可，性能可接受。                        |
| **1,000 ~ 10,000 行** | `COPY`        | **明显更快**  | 开始出现分水岭，`COPY` 的优势开始显现。           |
| **10,000 行以上**     | `COPY`        | **绝对碾压**  | 必须使用 `COPY`，否则插入时间会从秒级变成分钟级。 |

---

### 3. 什么时候【不该】用 `COPY`？

虽然 `COPY` 快，但在以下几种情况，`INSERT` 反而更好：

1. **需要 `ON CONFLICT` 处理**：

* `COPY` 不支持 `ON CONFLICT DO UPDATE`（Upsert）。如果你需要“存在则更新，不存在则插入”，`COPY` 无法直接实现。
* *技巧*：可以通过 `COPY` 到临时表 $\rightarrow$ 从临时表 `INSERT INTO ... ON CONFLICT` 到目标表。

2. **极小量数据**：

* 如果你只插入 5 行数据，建立 `COPY` 流的开销反而比直接执行一个 SQL 语句要大。

3. **需要触发器/复杂逻辑逐行处理**：

* 虽然 `COPY` 也会触发触发器，但某些极其复杂的每行计算在 SQL 层处理可能更灵活（尽管性能会下降）。

### 4. 终极性能优化建议

如果你在使用 `COPY` 处理数百万行数据，还可以尝试：

1. **暂时关闭索引**：在 `COPY` 之前删除索引，导入完成后重新创建索引（重建索引比边导入边更新索引快得多）。
2. **禁用外键检查**：临时禁用约束检查。
3. **使用二进制格式**：`COPY` 的 `BINARY` 格式比 `CSV` 格式更快，因为它省去了字符串到数字/日期的转换时间（但可读性差，调试难）。
